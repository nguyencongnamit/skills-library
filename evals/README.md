# `evals/` — Skills Library Evaluation Harness

A reproducible harness for measuring whether the security skills,
DLP patterns, vulnerability database, and MCP scanners actually help
when wired into a coding agent. The harness is designed so an
*outside* reviewer can clone this repo, run one command, and verify
every claim the project makes about precision, recall, and coverage.

## What lives here

```
evals/
├── README.md                                  # this file
├── run-evals.sh                               # top-level harness (no LLM required)
├── fixtures/
│   ├── secret-generation/                     # prompts that tempt an agent to hardcode secrets
│   ├── secret-detection/                      # pointer to skills/secret-detection/tests/corpus.json
│   ├── dependency-choice/                     # package.json / requirements.txt with planted bad deps
│   ├── cicd-hardening/                        # .github/workflows/*.yml with known anti-patterns
│   ├── auth-patterns/                         # JWT / OAuth / session anti-patterns
│   └── ssrf/                                  # SSRF sinks (user URL pass-through, metadata endpoint)
├── baselines/
│   ├── README.md                              # baseline schema
│   ├── secret-detection-static.md             # auto-regenerated by run-evals.sh
│   ├── no-instructions.json                   # template — populate with LLM run
│   ├── minimal-skill.json                     # template — populate with LLM run
│   └── full-mcp.json                          # template — populate with LLM run
└── benchmarks/
    └── secret-detection-vs-gitleaks.py        # precision/recall vs gitleaks default ruleset
```

## What this harness covers

| Category | Source-of-truth | Automated? | Baseline file |
| --- | --- | --- | --- |
| `secret-detection` (regex / DLP) | `skills/secret-detection/tests/corpus.json` | yes | `baselines/secret-detection-static.md` |
| `dependency-choice` | `vulnerabilities/supply-chain/malicious-packages/*.json` + `typosquat-db/known_typosquats.json` | partial (the static check is via `scan_dependencies` — wire-up is documented in each fixture's `expected.json`) | n/a |
| `cicd-hardening` | `skills/cicd-security/checklists/github_actions_hardening.yaml` | partial (driven by the P3 `scan_github_actions` MCP tool) | n/a |
| `auth-patterns` | `skills/auth-security/rules/*.json` | manual review per fixture | n/a |
| `ssrf` | `skills/ssrf-prevention/rules/*.json` | manual review per fixture | n/a |
| `secret-generation` (agent leaks secrets in code it writes) | each fixture's "Insecure response" block | requires an LLM | `baselines/{no-instructions,minimal-skill,full-mcp}.json` |

## Running it locally

```bash
./evals/run-evals.sh
```

The harness runs three checks, in order:

1. `skills-check test secret-detection` — replays the >230-fixture
   corpus against the rule set (built from `skills/secret-detection/
   rules/dlp_patterns.json`).
2. `python3 evals/benchmarks/secret-detection-vs-gitleaks.py` — runs
   the same corpus against **both** the skills-library regex set
   **and** [gitleaks](https://github.com/gitleaks/gitleaks) (if
   installed), and writes a Markdown precision/recall table to
   `evals/baselines/secret-detection-static.md`. The gitleaks column
   is the only one that exercises a ruleset that was *not* tuned
   against this corpus — see the "Honesty note" at the top of that
   file.
3. A static well-formedness check on every `expected.json` in
   `evals/fixtures/`: parse the JSON, then verify that every `source`
   field resolves to a real path in the repo. This catches link rot
   when a skill rule is renamed.

`SKIP_GITLEAKS=1 ./evals/run-evals.sh` runs steps 1 and 3 only.

## Why this harness exists

The critique that motivated `P5` was: *"You claim the skills improve
an agent's security behaviour. Show me the numbers."* That claim
breaks down into two sub-claims:

1. **Static precision/recall is competitive with general-purpose
   scanners.** Step 2 above puts that on paper.
2. **An agent given these skills makes safer choices than one without
   them.** This needs an LLM in the loop. The schema for that exists
   under `baselines/` so anyone with API access can produce a real
   measurement; we deliberately do not ship the LLM-driving code (see
   `AGENTS.md` — this project does not encourage AI-driven
   contribution).

## Adding new fixtures

1. Drop a new sample file (and its companion `expected.json`) into
   the right category under `evals/fixtures/`.
2. Cite the source rule / advisory in `expected.json` — every
   fixture must point at a real entry in `skills/` or
   `vulnerabilities/`. No invented threats.
3. Run `./evals/run-evals.sh` to make sure the fixture validates.
4. Send a PR.

## AI authorship disclosure

This harness was drafted with AI assistance per `AGENTS.md`. The
benchmark logic is a re-implementation of the existing Go corpus
runner (`cmd/skills-check/cmd/test.go`); the fixture files reuse
real OWASP / RFC / vendor examples cited inline; the baselines are
empty templates rather than synthetic results, because synthetic
baselines would be misleading.
